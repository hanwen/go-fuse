‚óŹ The bug, concisely:

  LOOKUP request and reply alias the same memory in the uring transport. On the
  request side, the kernel writes the lookup name into e.payload[0:nameLen]. On
  the reply side, the handler writes an EntryOut into
  e.payload[0:sizeof(EntryOut)] ‚ÄĒ same buffer.

  For a positive lookup that doesn't matter ‚ÄĒ setEntryOut overwrites every
  field. But for negative-entry replies (ENOENT with NegativeTimeout > 0),
  bridge.Lookup only calls out.SetEntryTimeout(...) and returns. Per the go-fuse
  contract, the other fields are supposed to stay zero. With uring, they don't:
  the kernel-written name bytes are still sitting at offset 0, and
  EntryOut.NodeId (uint64 at offset 0) reads as the first 8 bytes of the name.
  For ".Trash-1000\0", that's 0x312D68736172542E = ".Trash-1". The kernel takes
  that as a real NodeID, caches it, and panics us later with unknown node 
  3543603326575858734.

  The same mechanism is biting your EntryValid/AttrValid now (tE=3.4e10s, 
  tA=2.07e11s) ‚ÄĒ those are stale bytes from a previous reply, surviving into the
  partial-init path where the handler only sets Attr and not the validity
  timeouts.

  You're right that fixing it in bridge.Lookup is wrong direction ‚ÄĒ the uring
  transport is violating the API contract, not the bridge. I should revert that
  and fix it at the transport layer. The catch is: input name and output struct
  share e.payload, so I can't just clear(payload[:outSize]) before dispatch
  (that's what wiped .go-fuse-epoll-hack).

‚óŹ The architectural difference is what's biting you.

  Classic /dev/fuse path (server.go:660):
  go ms.handleRequest(req)
  One reader goroutine pulls requests from /dev/fuse and fans out every request
  to a fresh goroutine. The Go scheduler spreads handler work across all
  available P's. A 4-process find workload ends up with N handler goroutines
  running in parallel.

  Our uring path (uring_dispatch_linux.go:Run):
  for {
      ioUringEnter(...)            // submit + wait
      for cqe := range CQEs {
          q.handle(idx)            // ‚Üź SYNCHRONOUS, in the queue goroutine
      }
  }
  Each per-CPU queue goroutine processes CQEs inline, serially. With 4 entries
  per queue, the kernel has at most 4 requests parked in registered slots per
  CPU, but only one is being executed by Go code at any moment. The fan-out is
  implicit in the per-CPU queues (one goroutine per CPU), not per-request.

  So for a workload with many small, fast-handler ops (LOOKUP/GETATTR/READDIR)
  and lots of concurrent clients, classic happily uses every core through
  goroutine fan-out, while uring is throttled to ‚Č§ num_possible_cpus √ó 1 active
  handler. On a box where find waits more than it computes, uring is just adding
  overhead (per-request clears, the copy-back, an extra make for
  inBuf/inTogether) without the parallelism win.

  Where uring should win in this design as it stands:
  - Workloads dominated by syscall overhead rather than handler cost ‚ÄĒ e.g. lots
  of tiny ops where read/writev round-trips cost more than the FS handler
  itself. Splice already removed most of that for /dev/fuse, so the gap is
  narrow.
  - Reply-heavy paths once you stop disabling splice for /dev/fuse (e.g. small
  synchronous READs we can't zero-copy via uring). That's still a loss for
  uring, not a win.
  - Very high single-CPU op rate where the kernel's per-CPU queue avoids
  cross-CPU cache traffic. Hard to measure without taskset.

  Where the design needs to change to match classic on this kind of workload:
  1. Decouple the dispatch goroutine from the handler: have q.handle enqueue the
  request to a small worker pool (or just go q.dispatchAndCommit(idx)) and have
  the queue goroutine only do submit/poll. Then up to N handlers run
  concurrently per CPU instead of one.
  2. Bigger uringDefaultEntries (today:¬†4). With (1) in place, more parked slots
  = more inflight requests = more SQE batching per io_uring_enter.
  3. Get rid of the per-request make([]byte, 168) and make([]byte, iovLen(...))
  allocations (inBuf, inTogether). The classic path uses a pooled requestAlloc
  exactly for this.

  (1) is the big one ‚ÄĒ without it you're comparing N parallel handlers to
  ~`num_cpus` parallel handlers.

  Each reader, before handling its request inline, spawns a successor goroutine
  to read the next one if no other reader is waiting. So concurrency grows
  organically: under load you end up with as many concurrent (reader + inline
  handler) pairs as there is pending work, up to maxReaders. That's how classic
  gets goroutine fan-out without explicitly going go handle() per request.

  So per-request, classic and uring both run the handler inline on the
  dispatching goroutine. The real difference is how many of those goroutines 
  exist:
  - Classic: dynamic, grows with traffic up to maxReaders (~16+).
  - Uring: fixed at num_possible_cpus queue goroutines, each draining its own
  CQE stream serially.

  dest, calls req.setInput(dest[:n]), returns one request. No batching at the
  syscall layer; the fan-out comes from the successor-reader trick.
  - io_uring: N CQEs per io_uring_enter, where N ‚ąą [1, in-flight entries]. The
  Run loop drains all available CQEs before re-entering. So
  syscall-amortization-wise we're already winning per request ‚ÄĒ what we're
  losing is goroutine-level parallelism.

‚ĚĮ is there a disadvantage to increasing uringDefaultEntries hardcoded?          

‚óŹ Mostly memory. Each entry carries a payload buffer sized to negotiated
  MaxWrite (typically 128 KB, up to 1 MB), plus the header struct and op-out
  scratch. Total cost is roughly:

  entries √ó num_possible_cpus √ó (MaxWrite + ~600 B)

  On an 8-CPU box with MaxWrite=128KB:
  - entries=4 ‚Üí ~4 MB pinned
  - entries=32 ‚Üí ~32 MB
  - entries=128 ‚Üí ~128 MB

  With MaxWrite=1MB those go 8√ó up. The buffers are also kernel-pinned
  (copy_to_user targets) for the lifetime of the queue, so they sit in your RSS
  regardless of load.

  Other smaller costs:
  - Startup cost: one REGISTER SQE per entry at queue init. Linear, but trivial
  ‚ÄĒ microseconds.
  - Kernel-side fuse_ring_ent array: similar scaling on the kernel side. Not
  user-controlled.
  - Wasted slots on light workloads: most entries sit idle most of the time.
  Memory waste, not a correctness issue.

  The right knob isn't really entries alone ‚ÄĒ it's entries √ó payload_size. For
  metadata-heavy workloads (LOOKUP/GETATTR/STATFS) you don't need 128 KB per
  entry; 4 KB would do. So a smarter version is to make entry count and
  per-entry payload size separate parameters, then a metadata-tuned mount could
  do entries=64, payload=4KB (~2 MB total instead of 32 MB).

  For a quick experiment, just bumping uringDefaultEntries = 32 is fine on a
  developer machine ‚ÄĒ at default MaxWrite=128KB you're spending ~32 MB extra to
  test the hypothesis.

‚ĚĮ Why must there be a queue per CPU ? how can this scale to 192 CPU machines?   

‚óŹ It's a kernel-imposed constraint, not a design choice. From
  fs/fuse/dev_uring.c:

  1. Routing key is task_cpu(current) ‚ÄĒ when a FUSE request arrives, the kernel
  picks the queue based on which CPU the requesting task is currently on.
  There's no other routing knob exposed to userspace.
  2. is_ring_ready requires all qids registered ‚ÄĒ fuse_block_alloc blocks every
  new request until every qid in [0, num_possible_cpus) has at least one
  registered entry. If you skip qid 7 on a 192-CPU box, any task that gets
  scheduled on CPU 7 hangs at the first FUSE syscall, forever.

  So the queue count is dictated by num_possible_cpus(), not by what we'd
  choose. The userspace API has no way to say "I want N queues, route everything
  through them" ‚ÄĒ the kernel hardcodes per-CPU routing.

  192-CPU scaling cost in the current design:

  192 CPUs √ó 32 entries √ó (128 KB payload + ~600 B) ‚Čą 768 MB pinned

  Plus 192 io_uring fds, 192 LockOSThread goroutines, 192 entries in
  ms.uringQueues. Almost all of it idle on any real workload ‚ÄĒ at 80k req/s
  you'd touch maybe 8‚Äď16 queues actively.

  This is a real problem with the kernel ABI itself, not just our
  implementation. The kernel also pays a similar cost (per-queue fuse_ring_queue
  state, per-entry fuse_ring_ent). The libfuse reference implementation has the
  same shape and the same scaling cliff.

  Mitigations available to us (userspace):

  1. Right-size MaxWrite per workload. Metadata-only mounts don't need 128 KB
  per entry ‚ÄĒ at 4 KB you're at 24 MB total on 192 cores, manageable. We already
  negotiate MaxWrite at INIT; we could clamp it tighter when uring is on.
  2. Right-size uringDefaultEntries based on CPU count. Drop to 4 on big
  machines (192 cores √ó 4 entries √ó 128 KB = 96 MB) since high CPU count is
  itself the parallelism source ‚ÄĒ you don't also need deep per-queue queues.
  Make it min(32, max(4, 256/num_cpus)) or similar.
  3. Lazy registration with NUMA awareness ‚ÄĒ only fully register queues for CPUs
  we expect traffic on, but this conflicts with is_ring_ready (kernel blocks
  until all are ready). We'd have to register all 192 but use tiny placeholder
  entries (e.g. single 4 KB entry per low-traffic qid) and grow on demand.
  Doable but invasive.
  4. Smaller per-entry headers. Our fuseUringReqHeader (288 B) and opOutScratch
  (288 B) are small; the win is in the payload.

  Mitigations not available:

  - Fewer queues than CPUs. Kernel ABI doesn't permit.
  - Shared payload pool across queues. The iov registered at REGISTER time is
  the kernel's copy_to_user target; each entry must own its buffer.
  - Cross-CPU steering. The kernel hardcodes task_cpu(current); no way to
  redirect.

  So short answer: the design doesn't scale gracefully to 192 cores in its 
  current form, and that's structural ‚ÄĒ the only adjustment knobs we own are
  MaxWrite and entries. A 192-core mount with default settings would burn 768 MB
  to support a workload that might actively use 4 CPUs. The right product fix
  is probably (1)+(2): tune both based on CPU count, and document the resulting
  RSS.

‚ĚĮ in the current design, if we get 3 reqs in a single enter, how do we handle   
  them?                                                                         

‚óŹ Look at (*uringQueue).Run:

  for {
      ioUringEnter(q.ring.fd, pending, 1, _IORING_ENTER_GETEVENTS)
      pending = 0
      for {

  For 3 CQEs in one enter, the inner loop is strictly serial:

  1. peekCqe returns CQE for entry idx0 ‚Üí advanceCq ‚Üí q.handle(idx0) runs the 
  handler inline: copies bytes out of e.header/e.payload, dispatches through
  protocolServer.handleIov ‚Üí bridge ‚Üí user FS code ‚Üí writes reply into
  opOutScratch ‚Üí copies it into e.payload ‚Üí queues a COMMIT_AND_FETCH SQE into
  the SQ (not yet sent to kernel).
  2. peekCqe returns CQE for entry idx1 ‚Üí same thing, end-to-end.
  3. peekCqe returns CQE for entry idx2 ‚Üí same thing.
  4. peekCqe returns nil ‚Üí break.
  5. Back to ioUringEnter with pending=3, atomically submitting all three queued
  COMMIT_AND_FETCH SQEs and waiting for the next batch.

  Two consequences:

  Reply SQEs are batched (good): all three COMMIT_AND_FETCHes ride out on one
  enter syscall along with the next wait. That's why we see 4.70 CQE/enter.

  Handlers run sequentially in the queue goroutine (bad): while we're inside
  q.handle(idx0), CQEs 1 and 2 are already sitting in the CQ buffer ‚ÄĒ the kernel
  has copied their request data into e.header/e.payload, the work is ready ‚ÄĒ
  but we won't touch them until handler 0 returns. If handler 0 takes 17 ¬Ķs and
  there are 3 ready CQEs, the third one waits 34 ¬Ķs of pure queue-goroutine-busy
  time before it even gets parsed. That's the per-queue serialization ceiling.

  So "3 CQEs in one enter" = 3 sequential q.handle calls = 3 √ó 
  per-request-handler-latency of queue time before the goroutine sleeps again.
  The batching saves syscalls but not handler latency.

  This is the structural reason the previous run plateaued at ~80k req/s on 8
  queues even with 32 entries: per-queue throughput is 1 / handler_latency, and
  the only way past it without fanning handlers out to worker goroutines is to
  make handle() itself faster.

‚úĽ Cooked for 28s

‚ĚĮ in this case, must we return the  3 responses at the same time, or can they   
  complete separately?                                                          

‚óŹ They can complete separately, but with the current code they don't ‚ÄĒ and
  that's the design choice, not an io_uring constraint.

  How the kernel actually picks up our replies:

  Without IORING_SETUP_SQPOLL, the kernel only reads the submission queue when 
  userspace calls io_uring_enter. So a COMMIT_AND_FETCH SQE sitting in the SQ
  ring is invisible to the kernel until our next enter. Even though handler 0 is
  done and the reply bytes are in e.payload, the kernel won't deliver them to
  the requesting task until we enter again.

  In the current "drain all CQEs, then enter" loop, that means all three replies
  are released to their requesting tasks at the same enter call after the third
  handler returns. Total user-visible latency for request 0 ‚Čą 3 √ó handler_time,
  not 1 √ó handler_time. That's the structural cost.

  Three ways to deliver them separately:

  1. Enter per reply. After each q.handle(idx), call ioUringEnter(fd, 1, 0, 0)
  (non-blocking submit, no wait). The kernel sees the COMMIT_AND_FETCH
  immediately, the requesting task wakes up, then we loop back and pick up the
  next CQE. Cost: one extra syscall per request, which is exactly the cost we
  set up io_uring to avoid. Probably bad.
  2. Worker goroutines per request, async commit. The queue goroutine just
  drains CQEs and dispatches each one to a worker (go q.dispatch(idx)). The
  worker runs the handler, writes the reply, queues its own COMMIT_AND_FETCH SQE
  under a small mutex protecting the SQ, and bumps a pending-counter. The queue
  goroutine periodically calls enter(pending, 0, 0) (non-blocking) to flush, or
  piggybacks the flush on its next blocking wait. Cost: SQ mutex + goroutine
  churn, but the latency of reply 0 is now decoupled from handlers 1 and 2 ‚ÄĒ
  they run on different P's. This is the real fix to per-queue serial dispatch.
  The same change unlocks intra-queue concurrency.
  3. SQPOLL mode (IORING_SETUP_SQPOLL). The kernel spawns a thread that polls
  the SQ continuously. Userspace writes the SQE + memory-barrier'd tail bump and
  the kernel picks it up within microseconds. Cost: one kernel thread per ring
  (so num_possible_cpus extra kernel threads on a 192-core box), plus the kernel
  thread spins for a configurable idle window before sleeping. Big-iron servers
  do use it, but it's a heavy footgun. Combined with our per-CPU queues, the
  kernel-thread cost is steep.

  (2) is the right answer for the workload you've been measuring. (3) is mainly
  interesting if you've already done (2) and want to remove the remaining enter
  syscalls entirely. (1) is a strict downgrade and only worth doing as a
  one-line test to confirm where the latency is.

  The fact that the current loop batches replies isn't required by io_uring ‚ÄĒ
  it's just the simplest dispatch shape, and it happens to align well with
  bursty traffic (enter-once-per-batch is fine when the batch is what you waited
  for anyway). For latency-sensitive single-request workloads, decoupling reply
  submission from handler ordering is the structurally correct move.
